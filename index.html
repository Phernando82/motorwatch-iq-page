Real industrial hardware — production environment

Hardware industrial real — entorno de producción

The system monitors vibration and temperature of four industrial motors in real time. Each motor is equipped with an IFM VVB306 IO-Link sensor measuring velocity (mm/s RMS), acceleration (g peak), and temperature (°C). The sensors connect to an IFM AL1901 IO-Link master, which communicates with a Siemens S7-1200 CPU 1214C DC/DC/DC via PROFINET. A Siemens MTP700 Unified Basic HMI provides operator interaction directly on the shop floor.

The operator can configure pre-alarm and alarm thresholds individually for each motor — velocity, acceleration, and temperature — directly from the HMI settings screen, protected by PIN authentication. Configuration changes are saved to retentive data blocks via a state machine (FB_Save_Parameters), ensuring parameters survive power cycles. The main screen displays live readings for all four motors grouped by production zone.

Visual alarm logic: Motors in normal state display in blue with white value fields. A pre-alarm turns the motor icon yellow and highlights the specific parameter field that triggered it, so the operator immediately knows which measurement exceeded the threshold. An alarm turns the motor icon red with the critical field highlighted in red. A visible pre-alarm LED output activates on the control panel.

Alarm response logic: On alarm, a zone-specific siren activates and a motor stop output can be enabled per motor configuration. An authenticated operator can permanently silence the alarm. An operator without full access can silence temporarily — the siren resumes after one hour if the fault has not been resolved. The output only re-enables after the root cause is cleared, preventing inadvertent restart.

El sistema monitoriza vibración y temperatura de cuatro motores industriales en tiempo real. Cada motor dispone de un sensor IFM VVB306 IO-Link que mide velocidad (mm/s RMS), aceleración (g pico) y temperatura (°C). Los sensores se conectan a un maestro IO-Link IFM AL1901, que se comunica con un Siemens S7-1200 CPU 1214C DC/DC/DC mediante PROFINET. El operario puede configurar individualmente los umbrales de prealarma y alarma para cada motor — velocidad, aceleración y temperatura — desde la pantalla de configuración de la HMI, protegida por autenticación PIN. Los cambios se guardan en bloques de datos retentivos mediante una máquina de estados (FB_Save_Parameters), asegurando que los parámetros sobreviven ciclos de alimentación. La pantalla principal muestra lecturas en tiempo real de los cuatro motores agrupados por zona de producción.

Lógica de alarma visual: Los motores en estado normal se muestran en azul con campos de valor en blanco. Una prealarma pone el icono del motor en amarillo y resalta el campo del parámetro específico que la activó, para que el operario identifique inmediatamente qué medición superó el umbral. Una alarma pone el motor en rojo con el campo crítico resaltado en rojo.

Lógica de respuesta a alarma: En alarma, se activa una sirena por zona y se puede habilitar la parada del motor por configuración. Un operario autenticado puede silenciar permanentemente. Sin acceso completo, el silenciamiento es temporal — la sirena reanuda tras una hora si el fallo no se ha resuelto. La salida solo se reactiva tras solucionar la causa raíz.

PySide6 desktop application

Aplicación de escritorio PySide6

The entire MotorWatch IQ stack is controlled from a single PySide6 desktop application. Rather than running services manually from terminals, all subprocesses — Mosquitto MQTT broker, InfluxDB writer, and the analytics engine — are managed as supervised child processes via subprocess.Popen, with stdout piped to the launcher console in real time. Process termination uses taskkill /F /T /PID to cleanly kill the full process tree on Windows.

The simulator tab allows injecting realistic fault scenarios per motor — thermal drift, vibration drift, impact spike, device failure — without requiring physical hardware. This simulator-first approach enabled developing and validating the entire analytics pipeline (M1–M8) before PLC integration. Each motor card shows live published values with AUTO/MANUAL mode and a configurable fault scenario selector.

Design decisions: Qt signals/slots for thread-safe UI updates from subprocess readers; QThread workers for InfluxDB queries to prevent UI freezing; Catppuccin Mocha palette applied consistently across all tabs and the generated HTML reports for visual cohesion; status LED dots driven by stdout parsing of subprocess output.

Todo el stack de MotorWatch IQ se controla desde una única aplicación de escritorio PySide6. En lugar de ejecutar servicios manualmente desde terminales, todos los subprocesos — broker MQTT Mosquitto, escritor InfluxDB y motor analítico — se gestionan como procesos hijo supervisados mediante subprocess.Popen, con stdout redirigido a la consola del launcher en tiempo real.

La pestaña simulador permite inyectar escenarios de fallo realistas por motor — deriva térmica, deriva de vibración, pico de impacto, fallo de dispositivo — sin necesidad de hardware físico. Este enfoque simulator-first permitió desarrollar y validar todo el pipeline analítico antes de la integración con el PLC.

Decisiones de diseño: señales/slots Qt para actualizaciones seguras del hilo UI; workers QThread para consultas InfluxDB; paleta Catppuccin Mocha aplicada consistentemente en todas las pestañas y los informes HTML generados.

Industrial IoT data pipeline

Pipeline de datos Industrial IoT

OPC UA (IEC 62541) was chosen as the primary PLC interface. Unlike Snap7 which uses an undocumented Siemens protocol, OPC UA is the Industry 4.0 standard for machine-to-machine communication, natively supported on S7-1200 firmware ≥ 4.1 without additional hardware. The implementation uses asyncua with async/await and DataChange subscriptions — the PLC pushes updates rather than the collector polling, reducing network overhead and latency.

Snap7 is implemented as a hot-standby fallback. If the OPC UA connection fails, the collector automatically switches to Snap7 direct S7 communication using the same exponential backoff reconnection logic (ceiling 60 s). The MQTT payload schema is identical for both paths, making the switch transparent to all downstream consumers.

InfluxDB 2.7 with Flux query language was chosen over InfluxDB 1.x for its native support for windowed aggregations, data downsampling tasks, and the influxdb-client-python SDK with CallbackAPIVersion.VERSION2. Tags (motor_id) enable efficient per-motor filtering without scanning all series.

Se eligió OPC UA (IEC 62541) como interfaz PLC principal. A diferencia de Snap7, que usa un protocolo propietario de Siemens, OPC UA es el estándar de Industry 4.0 para comunicación máquina a máquina, soportado de forma nativa en firmware S7-1200 ≥ 4.1 sin hardware adicional. La implementación usa asyncua con async/await y suscripciones DataChange — el PLC envía actualizaciones en lugar de que el colector haga polling.

Snap7 se implementa como fallback en caliente. Si la conexión OPC UA falla, el colector cambia automáticamente a comunicación directa S7 con la misma lógica de reconexión con backoff exponencial (techo 60 s). El esquema de payload MQTT es idéntico para ambas rutas, haciendo el cambio transparente para todos los consumidores downstream.

InfluxDB 2.7 con Flux se eligió por su soporte nativo de agregaciones por ventana, tareas de downsampling, y el SDK influxdb-client-python. Los tags (motor_id) permiten filtrado eficiente por motor sin escanear todas las series.

Grafana dashboards & maintenance reports

Dashboards Grafana e informes de mantenimiento

Two Grafana 13 dashboards provide real-time and analytical visibility. The sensor dashboard (MotorWatch IQ) shows alarm state timelines for all four motors, current v-RMS and a-Peak stat panels with ISO 20816-3 colour thresholds, temperature gauges, and a v-RMS time series sparkline — refreshed every 5 seconds. The anomaly dashboard (Motor Anomaly) shows Isolation Forest scores, alert level state timelines, trend slopes in mm/s/h, and a CWRU-based fault classification table with fault probability bars.

The Grafana sensor data model uses both numeric fields (alarm_state_num 0–3) and string fields (alarm_state_str) written to the motor_telemetry measurement. The State Timeline panel uses threshold-based numeric colouring, while string fields enable human-readable tooltips. Sentinel values (−1.0) are written for fault/offline states rather than dropping the point, preserving timeline continuity.

Dos dashboards de Grafana 13 proporcionan visibilidad en tiempo real y analítica. El dashboard de sensores muestra líneas temporales de estado de alarma, paneles de estadísticas de v-RMS y a-Pico con umbrales de color ISO 20816-3, indicadores de temperatura y sparklines de series temporales — actualizado cada 5 segundos. El dashboard de anomalías muestra puntuaciones Isolation Forest, líneas de nivel de alerta, pendientes de tendencia en mm/s/h y una tabla de clasificación de fallos basada en CWRU.

El modelo de datos usa tanto campos numéricos (alarm_state_num 0–3) como campos de cadena (alarm_state_str). Se escriben valores sentinel (−1.0) para estados de fallo/offline en lugar de descartar el punto, preservando la continuidad de la línea temporal.

Machine learning analytics pipeline

Pipeline analítico de machine learning

The analytics engine (anomaly_detector.py) runs as a persistent supervised process, reading time-windowed sensor data from InfluxDB every 30 seconds and writing results to the motor_anomaly measurement. The pipeline combines three independent analytical methods: unsupervised anomaly detection, statistical trend analysis, and supervised fault classification.

The Isolation Forest model is trained on the CWRU (Case Western Reserve University) bearing dataset, a widely-used benchmark for rotating machinery fault detection. Training produces a model per motor that scores incoming windows from 0.0 (normal) to 1.0 (anomalous). No labelled fault data from the target machines is required — the model learns the normal operating envelope and scores deviations from it.

The trend analyser applies linear regression to rolling v-RMS windows, computing slope in mm/s/h and projecting ETA to pre-alarm and alarm thresholds. This transforms reactive alarming into predictive maintenance: the system can warn of a developing fault hours before the threshold is breached. Alert hysteresis prevents flapping between states on noisy signals.

The fault classifier uses frequency-domain features from the CWRU dataset to identify fault type — normal, inner race fault, outer race fault, or ball fault — with a probability score. The generated maintenance report combines all three outputs into a prioritised intervention plan with specific recommended actions per fault type, following ISO 20816-3 severity categories.

El motor analítico (anomaly_detector.py) se ejecuta como proceso supervisado persistente, leyendo datos de sensores por ventana temporal de InfluxDB cada 30 segundos y escribiendo resultados en la medición motor_anomaly. El pipeline combina tres métodos analíticos independientes: detección de anomalías no supervisada, análisis de tendencia estadístico y clasificación supervisada de fallos.

El modelo Isolation Forest se entrena con el dataset CWRU (Case Western Reserve University), un benchmark ampliamente utilizado para detección de fallos en maquinaria rotativa. No se requieren datos de fallo etiquetados de las máquinas objetivo — el modelo aprende la envolvente de operación normal y puntúa las desviaciones.

El analizador de tendencias aplica regresión lineal a ventanas deslizantes de v-RMS, calculando la pendiente en mm/s/h y proyectando el ETA hasta los umbrales de prealarma y alarma. El clasificador de fallos utiliza características en el dominio de la frecuencia del dataset CWRU para identificar el tipo de fallo con una puntuación de probabilidad.
